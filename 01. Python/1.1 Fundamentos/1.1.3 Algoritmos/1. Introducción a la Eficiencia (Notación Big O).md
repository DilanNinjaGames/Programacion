### 1.1 ¿Por qué no medimos el tiempo en segundos?

  

Imagina que escribes un algoritmo para ordenar una lista de 1.000 nombres y lo pruebas en tu computadora: tarda 0.3 segundos. Tu amigo ejecuta el mismo código en una máquina del año 2010 y le tarda 4 segundos. Un servidor de Google lo ejecuta en 0.002 segundos.

  

**¿El algoritmo es rápido o lento?** No puedes saberlo midiendo en segundos, porque el tiempo depende del hardware, del sistema operativo, de otros procesos corriendo en paralelo, e incluso de la temperatura del procesador.

  

Necesitamos una medida **independiente de la máquina**. La solución: en lugar de medir tiempo, **contamos el número de pasos (operaciones) que ejecuta el algoritmo** en función del tamaño de la entrada `n`.

  

La **Notación Big O** describe cómo crece el número de operaciones cuando `n` crece hacia el infinito. No nos interesa el número exacto de pasos (eso depende de detalles de implementación), sino el **patrón de crecimiento**: ¿si duplico la entrada, cuánto más trabajo hace el algoritmo?

  

> **Big O describe el peor escenario posible.** Es una garantía de rendimiento: "este algoritmo *nunca* será peor que esto."

  

```python

# ── ¿Qué contamos como "un paso"? ─────────────────────────────────────────

# Contamos operaciones elementales: comparaciones, asignaciones, accesos a memoria.

# Ignoramos constantes porque en el largo plazo no cambian el patrón de crecimiento.

  

# Ejemplo concreto: buscar el número 7 en una lista de n elementos

lista = [3, 1, 9, 5, 7, 2, 8, 4, 6]

  

# En el MEJOR caso (7 está en posición 0): 1 comparación → O(1)

# En el PEOR caso (7 está al final o no está): n comparaciones → O(n)

  

# Ignoramos las constantes:

# Si un algoritmo hace 3n + 100 pasos, su Big O es O(n)

# porque cuando n=1.000.000, el +100 es irrelevante (0.00001% del total)

  

# Si un algoritmo hace 5n² + 3n + 200 pasos, su Big O es O(n²)

# porque n² domina a n cuando n crece

```

### 1.2 Complejidades Comunes Explicadas con Manzanas

  

Usemos una analogía concreta: tienes una caja con `n` manzanas y quieres realizar distintas operaciones sobre ellas.

  

---

  

#### $O(1)$ — Constante: "Sé exactamente dónde está"

  

**Analogía:** La caja tiene una etiqueta que dice cuántas manzanas hay. Leer la etiqueta tarda lo mismo sin importar si hay 10 o 10 millones de manzanas.

  

**En código:** Acceder a un elemento de una lista por su índice. Python sabe la dirección exacta en memoria porque las listas almacenan punteros contiguos.

  

```python

frutas = ["manzana", "pera", "uva", "kiwi", "mango"]

  

# O(1): Python calcula la dirección = dirección_base + (índice × tamaño_puntero)

# No importa si la lista tiene 5 o 5 millones de elementos: es UNA operación

primera  = frutas[0]      # 1 operación

ultima   = frutas[-1]     # 1 operación

cualquiera = frutas[3]    # 1 operación

  

# También son O(1):

longitud = len(frutas)    # Python guarda el tamaño como atributo del objeto

existe   = "pera" in {    # Los sets usan hashing: búsqueda O(1) promedio

    "manzana", "pera", "uva"

}

  

# Gráfica mental:

# n=10      → 1 paso

# n=1,000   → 1 paso

# n=1,000,000 → 1 paso

# La línea en la gráfica es HORIZONTAL: no crece nunca

```

  

---

  

#### $O(\log n)$ — Logarítmica: "Elimino la mitad en cada paso"

  

**Analogía:** Tienes un diccionario de 1.000 páginas y buscas la palabra "Python". No lees desde la página 1. Abres por la mitad (página 500). Si "Python" va después de la letra P (sí), descartás las primeras 500 páginas. Abres la mitad del rango restante (página 750). Y así. En unos 10 pasos, encontrás la palabra.

  

**La clave:** $\log_2(1.000) \approx 10$. Con $n = 1.000.000.000$ elementos, solo necesitas $\approx 30$ pasos.

  

```python

# O(log n): cada paso DIVIDE el problema a la mitad

# El logaritmo base 2 responde: "¿cuántas veces puedo dividir n entre 2?"

  

import math

  

def pasos_binaria(n):

    """Cuántas comparaciones hace la búsqueda binaria en el peor caso"""

    return math.ceil(math.log2(n))

  

print(f"n=8:           {pasos_binaria(8)} pasos")         # 3 pasos

print(f"n=1,000:       {pasos_binaria(1_000)} pasos")     # 10 pasos

print(f"n=1,000,000:   {pasos_binaria(1_000_000)} pasos") # 20 pasos

print(f"n=1,000,000,000:{pasos_binaria(1_000_000_000)} pasos")  # 30 pasos

  

# ¡Pasar de 1 millón a 1.000 millones de elementos

# solo agrega 10 pasos más! Eso es el poder de O(log n).

  

# Gráfica mental:

# n=1         → 0 pasos

# n=8         → 3 pasos

# n=1,000     → 10 pasos

# n=1,000,000 → 20 pasos

# La curva crece MUY lentamente: casi plana para n grandes

```

  

---

  

#### $O(n)$ — Lineal: "Reviso cada manzana una vez"

  

**Analogía:** Quieres encontrar la manzana más grande de la caja. No hay más remedio: tienes que revisar **cada una** al menos una vez. Si la caja tiene el doble de manzanas, tardarás el doble.

  

```python

# O(n): el número de operaciones crece proporcionalmente con n

  

def encontrar_maximo(lista):

    """

    Recorre cada elemento exactamente UNA vez.

    n=100 → ~100 comparaciones

    n=1,000,000 → ~1,000,000 comparaciones

    """

    if not lista:

        raise ValueError("Lista vacía")

  

    maximo_actual = lista[0]          # 1 operación

  

    for elemento in lista[1:]:        # n-1 iteraciones → O(n)

        if elemento > maximo_actual:  # 1 comparación por iteración

            maximo_actual = elemento  # 1 asignación (en el peor caso)

  

    return maximo_actual

  

numeros = [3, 7, 1, 9, 4, 6, 8, 2, 5]

print(encontrar_maximo(numeros))  # 9

  

# ¿Por qué no es O(2n) o O(3n)?

# Porque ignoramos las constantes. Lo que importa es que "si n se duplica,

# el trabajo se duplica". El coeficiente (1, 2, 3...) no cambia eso.

  

# Gráfica mental: línea recta diagonal

# n=100       → 100 pasos

# n=1,000     → 1,000 pasos

# n=1,000,000 → 1,000,000 pasos

```

  

---

  

#### $O(n^2)$ — Cuadrática: "Para cada manzana, reviso todas las demás"

  

**Analogía:** Quieres saber si hay dos manzanas del mismo tamaño exacto en la caja. Tomas la primera y la comparas con todas las demás (n-1 comparaciones). Luego tomas la segunda y la comparas con todas las demás (n-1). Y así para cada una. En total: $n \times (n-1) \approx n^2$ comparaciones.

  

```python

# O(n²): el bucle EXTERNO corre n veces, el INTERNO también n veces

# → n × n = n² operaciones totales

  

def hay_duplicados_cuadratico(lista):

    """

    Versión LENTA: compara cada par de elementos.

    Intencionalmente ineficiente para ilustrar O(n²).

    """

    n = len(lista)

    comparaciones = 0   # contador pedagógico

  

    for i in range(n):              # bucle externo: n iteraciones

        for j in range(i + 1, n):  # bucle interno: n-i-1 iteraciones

            comparaciones += 1

            if lista[i] == lista[j]:

                print(f"  → {comparaciones} comparaciones hechas")

                return True, comparaciones

  

    return False, comparaciones

  

# Demostración del crecimiento explosivo:

import random

for tamano in [10, 100, 1_000]:

    lista_sin_dupl = list(range(tamano))   # sin duplicados → peor caso

    _, comps = hay_duplicados_cuadratico(lista_sin_dupl)

    print(f"n={tamano:6d} → {comps:,} comparaciones")

  

# n=10    →        45 comparaciones

# n=100   →     4,950 comparaciones

# n=1,000 →   499,500 comparaciones  ← 100x más datos, ~10,000x más trabajo

  

# VERSIÓN EFICIENTE O(n): usando un set (búsqueda O(1) por hashing)

def hay_duplicados_eficiente(lista):

    vistos = set()

    for elemento in lista:

        if elemento in vistos:   # O(1) promedio

            return True

        vistos.add(elemento)

    return False

```
