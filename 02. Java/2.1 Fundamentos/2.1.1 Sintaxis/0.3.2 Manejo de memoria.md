### Manejo de Memoria: Heap vs Stack

#### 🔄 Contraste de Sintaxis

En Python **no piensas en esto**: el GC y el modelo de objetos uniforme lo manejan todo. En Java, la memoria está dividida en dos regiones con reglas distintas, y **confundirlas produce bugs de comparación muy difíciles de detectar**.

```
STACK (Pila)                    HEAP (Montículo)
────────────────────────        ────────────────────────────────
- Variables locales             • Todos los objetos (new ...)
- Tipos primitivos              • Arrays
- Referencias a objetos*        • Strings (String Pool)
- Parámetros de métodos         • Variables de instancia

- Tamaño fijo en compilación    • Tamaño dinámico en runtime
- Acceso: O(1), muy rápido      • Acceso: más lento (puntero)
- Se libera al salir el método  • Se libera por Garbage Collector
- StackOverflowError si se llena• OutOfMemoryError si se llena
```

**La distinción crítica — tipos primitivos vs objetos:**

java

```java
// En el STACK (valor directo):
int    a = 42;       // el valor 42 vive en el stack
double x = 3.14;     // el valor 3.14 vive en el stack
boolean flag = true; // el valor true vive en el stack

// En el HEAP (referencia en stack → objeto en heap):
String  s = "Hola";      // stack tiene referencia → heap tiene el objeto String
int[]   arr = {1,2,3};   // stack tiene referencia → heap tiene el array
MiClase obj = new MiClase(); // stack tiene referencia → heap tiene el objeto
//      ↑
//  La variable 'obj' en el stack apunta a la dirección en el heap
```

#### 💻 Bloque de Código — Demostración visual Stack vs Heap

java

```java
// MemoriaDemo.java
// Muestra por qué == no funciona igual con primitivos que con objetos

public class MemoriaDemo {

    // Este método crea su propio frame en el Stack al ser llamado
    public static void demostrarStack() {
        // Estas variables viven en el Stack frame de este método
        int x = 100;     // valor 100 directamente en el stack
        int y = x;       // COPIA del valor — x e y son independientes
        y = 200;

        // x no cambió porque y es una COPIA del valor (no una referencia)
        System.out.println("Stack — x: " + x + ", y: " + y); // x:100, y:200
        // Al terminar el método, este frame desaparece del Stack automáticamente
    }

    public static void demostrarHeap() {
        // Ambas variables en el Stack, pero apuntan al MISMO objeto en el Heap
        int[] arr1 = {1, 2, 3};   // arr1 (stack) → [1,2,3] (heap, dirección 0xA1)
        int[] arr2 = arr1;         // arr2 (stack) → [1,2,3] (heap, misma 0xA1)

        arr2[0] = 99;              // modifica el objeto en el heap

        // arr1[0] también cambió porque arr1 y arr2 apuntan al MISMO heap object
        System.out.println("Heap — arr1[0]: " + arr1[0]); // 99 ← ¡sorpresa!

        // Para copiar un array de verdad, necesitas Arrays.copyOf():
        int[] arr3 = java.util.Arrays.copyOf(arr1, arr1.length); // objeto NUEVO en heap
        arr3[0] = 777;
        System.out.println("Heap — arr1[0] tras copiar: " + arr1[0]); // sigue 99 ✅
    }

    public static void main(String[] args) {
        demostrarStack();
        demostrarHeap();

        // La trampa del String: == compara REFERENCIAS, no contenido
        String s1 = new String("Java"); // objeto nuevo en Heap
        String s2 = new String("Java"); // OTRO objeto nuevo en Heap
        String s3 = "Java";             // String Pool (reutiliza)
        String s4 = "Java";             // String Pool (misma referencia que s3)

        System.out.println(s1 == s2);       // false — referencias distintas
        System.out.println(s3 == s4);       // true  — misma referencia del Pool
        System.out.println(s1.equals(s2));  // true  — compara CONTENIDO ✅
    }
}
```

#### 🪲 "Rompe el Código"

java

```java
// ❌ ERROR MUY COMÚN: usar == para comparar Strings (viene del hábito Python)

public class BugComparacion {
    public static void main(String[] args) {

        String entrada = new String("admin");   // simula input del usuario
        String esperado = "admin";

        // ❌ El pythonista escribe esto (en Python == siempre compara contenido):
        if (entrada == esperado) {
            System.out.println("Acceso concedido");
        } else {
            System.out.println("Acceso denegado");  // ← siempre llega aquí
        }
    }
}

// ❌ POR QUÉ FALLA:
// 'entrada' fue creado con 'new String(...)' → objeto nuevo en el Heap (0xA1)
// 'esperado' viene del String Pool → dirección diferente (0xB3)
// == compara DIRECCIONES de memoria, no el contenido "admin"
// 0xA1 != 0xB3 → false, aunque el texto sea idéntico

// ✅ REGLA DE ORO JAVA:
// Para primitivos (int, double, boolean...): usa ==
// Para objetos (String, arrays, tus clases...): usa .equals()

if (entrada.equals(esperado)) {          // ✅ compara contenido
    System.out.println("Acceso concedido");
}

// BONUS — si entrada pudiera ser null, protégete así:
if ("admin".equals(entrada)) {           // ✅ null-safe: el literal nunca es null
    System.out.println("Acceso concedido");
}
```
